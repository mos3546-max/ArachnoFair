# ArachnoFair
新規ゲーム「アラクノフェア」AntigravityCLI版 開発設計書
1. プロジェクト概要
   本ドキュメントは、テキストベースのCLI環境「AntigravityCLI」において、高度な戦略性と心理的ジレンマを提供するカードゲーム「アラクノフェア」の論理構造および実装指針を定義するものである。
   コンセプト:  蜘蛛の巣状の動的フィールドを舞台にした、リソース管理と「数当て」ロジックによる探索型数理ゲーム。
   ターゲットプラットフォーム:  AntigravityCLI。低リソース環境での動作を前提とし、グリッド移動とテキストログによる高速な応答速度に最適化する。
   基本勝利条件:  最終エリア中央の「宝物庫」に到達し、キーアイテム「蜘蛛の金糸」を用いて解錠条件を満たし、宝を入手すること。
2. キャラクター・ロール（役職）仕様
   プレイヤーは開始時に以下の5つの役職から1つを選択する。各役職はステータスに加え、探索や戦闘に直結する固有の特性を持つ。
   役職別ステータス・能力表
   役職,移動力,アイテム発見率,戦闘力,特殊能力
   冒険家,高,高,低,探索特化。移動フェーズの選択肢が常に1つ多い。
   エンジニア,中,中,低,アイテム使用回数+1。罠マスのペナルティを無効化する。
   トレジャーハンター,低,中,高,セットコレクションの補正倍率強化。戦闘勝利確率が極めて高い。
   石油王,中,中,中,他プレイヤーと同一マスに停止時、資金（リソース）を奪取。
   魔女,高,低,高,呪い（相手の移動力を1に固定、またはバースト上限を一時的に-500）。
   プレイスタイルの戦略的分類
   探索・回避型（冒険家・エンジニア）:  高い機動力と罠耐性を活かし、戦闘を避けて「蜘蛛の金糸」を早期に収集する。
   制圧・強奪型（トレジャーハンター・石油王・魔女）:  高い戦闘力や特殊能力で他者を妨害し、後発ながら効率的にリソースを奪い取る。魔女は「被ターゲット率が高い」というリスクを戦闘力でカバーする必要がある。
3. フィールド構造：蜘蛛の巣状マップ
   マップは、中央の「宝物庫」に向かって複数の同心円状の経路が収束する「蜘蛛の巣状トポロジー」を採用する。
   アイテムマス:  ロール特性に基づき、カードまたはリソースの獲得判定を行う。
   罠マス:  通過時にHP減少。エンジニア以外は移動力が次ターン終了まで半減する。
   イベントマス:  「蜘蛛の金糸」の獲得、または「2進法ヒント」の取得機会。
   移動補正マス:  蜘蛛の糸による「粘着（移動-2）」や「風（特定方向へ+2）」の効果。
4. 戦闘システム：バースト・メカニクス
   戦闘およびカードプレイの基盤には、数理的裏付けを持つ「バースト（過積載）」システムを導入する。
   数値設定の設計思想
   Source Context（Legend of Stars）に基づき、ライフを「3000」、基礎ステータスを「100の倍数」に設定する。これは日本の通貨単位（金銭感覚）に準拠させることで、直感的な可読性と「大きな数字を扱う爽快感」をプレイヤーに与えるためである。
   バースト戦闘ロジック
   攻撃側は手札を累積させて攻撃値を高めるが、常に「許容上限（バースト値）」のジレンマに直面する。
   累積ペナルティ:  攻撃値が個別のバースト値を超えた瞬間、自滅ダメージ（HP-500）と行動終了が発生する。
   「5本のきゅうり」型終盤設計:  探索の最終段階（またはトリックの最後）では、手札に残った「高い数字」を出さざるを得ない状況が発生する。この際、バースト値を超えないように道中でいかに低い数字を温存するかが鍵となる。
5. 核心メカニクス：「蜘蛛の金糸」と2進法アクセスルール
   本ゲームの最大の特徴は、宝物庫の開放条件が「2進法的な数当てゲーム」のロジックで変動する点にある。
   必要数変動アルゴリズム（1〜15の範囲）
   宝物庫には「8・4・2・1」の重みを持つ4つの鍵穴（スロット）が存在する。開放に必要な金糸の総量  $N$  は、以下のロジックで算出・判定される。
   ターゲット数値の決定:  基礎値（1〜15）に対し、役職（魔女・石油王は+3）や所持アイテム（セット完成で-2）による補正を加味した動的な目標値が設定される。
   バイナリ・チェック・イベント:  プレイヤーは道中で「どの桁（8, 4, 2, 1）がONか」の断片的なヒントを得る。
   例：ヒント「1枚目と4枚目のカード（1と8）に含まれる数」→ 目標値は  $9 (1001_2)$ 。
   解錠プロセス:  宝物庫到達時、所持している「蜘蛛の金糸」を各スロットに割り当てる。合計値が目標値と一致した場合のみ解錠に成功する。
   UIによる情報提示（ドミニオン式デザインの応用）
   CLI上では、数字の表示形式をその機能に応じて使い分ける。
   総量表示（蓄積）:  金糸の所持数は「5」のように枠の外に数字が浮き出ているような強調表示を行い、得点トラック的な「蓄積」を視覚化する。
   用途表示（消費）:  各スロットへの投入数は「( 1 )」のように枠内に収まる表示を行い、リソースとしての「消費」を意識させる。
6. UI/UX設計（AntigravityCLI対応）
   画面を「マップ」「ステータス」「ログ」の3セクションに固定し、テキストのみで状況を伝える。
   コミュニケーション制限と情報の曖昧化
   Source Context（The Game / ito）の知見を活かし、協力・対戦における情報の透明度を意図的に下げる。
   形容詞による通知:  自身の正確なバースト値や金糸の目標値は数値で表示されない。
   目標値への遠さ：「絶望的」「かなり必要」「もう少し」「あと一歩」
   バーストの危険度：「安全」「微熱」「過熱」「臨界点」
   意図:  具体的な数字を言わずに価値観を共有する（ito）もどかしさを、CLIのテキストログを通じて戦略的な駆け引きに変換する。
7. 数理的整合性と「不必要な壁」の定義
   確率の罠への対策
   人間が「3連続で引けない」ことを理不尽に感じる脳の特性（確率の罠）を考慮し、ドロー時には「擬似乱数補正（連続してハズレを引く確率を下げる処理）」を実装する。
   ゲームの本質的体験
   バーナード・スーツの定義に従い、本ゲームにおける「不必要な壁」を「宝物庫に直行せず、あえて2進法のパズル（金糸収集と桁の特定）を解く遠回り」と定義する。 この効率的でない営みこそが、プレイヤーが最終的に宝を手にした際の達成感を最大化させるためのアーキテクチャである。
8. 実装上の禁止・制約事項
   Markdown準拠:  視覚的補助にMermaidコードブロック、画像、複雑なグラフを使用することを禁止する。
   テーブル構造:  全データは標準的なMarkdownテーブルを用いて提示し、開発者が即座に配列データとして定義できるように構成すること。
   UI制約:  すべての表示は120文字×30行程度のテキストウィンドウ内に収まる情報量に制御すること。
